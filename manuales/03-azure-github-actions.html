Harmony v3 · CI/CD Automático

Azure + GitHub Actions
— Despliegue automático

Cada push a main compila y despliega automáticamente en la VM de Azure. Cero intervención manual.

~35 min setup inicial Ubuntu 22.04 LTS GitHub Actions

Cómo funciona el pipeline

Push a main
→
GitHub Actions
→
SSH a Azure VM
→
git pull + build + restart

La estrategia es pull + build en el servidor: GitHub Actions hace SSH al servidor y ejecuta un script que descarga el código nuevo, compila Go, compila React, y reinicia el servicio. Simple y sin artefactos cross-compilation.

Paso 1 — Preparar la VM de Azure

Primero asegúrate de tener la VM funcionando con todas las dependencias. Si ya tienes el despliegue manual listo, salta al Paso 2. Si es una VM nueva, instala lo siguiente (como root/sudo):

# Instalar dependencias (igual que en el manual de Azure)
apt update && apt upgrade -y
apt install -y git curl wget build-essential nginx certbot python3-certbot-nginx redis-server

# Go 1.22
wget https://go.dev/dl/go1.22.5.linux-amd64.tar.gz
tar -C /usr/local -xzf go1.22.5.linux-amd64.tar.gz
echo 'export PATH=$PATH:/usr/local/go/bin' > /etc/profile.d/go.sh

# Node.js 20
curl -fsSL https://deb.nodesource.com/setup_20.x | bash -
apt install -y nodejs

# PostgreSQL 16
/usr/share/postgresql-common/pgdg/apt.postgresql.org.sh -y
apt install -y postgresql-16
systemctl enable --now postgresql redis-server

Paso 2 — Estructura del servidor

1
Clonar el repositorio en el servidor

El pipeline usará git pull para actualizar el código. El repositorio debe estar clonado en el servidor.

# Crear estructura
useradd --system --no-create-home --shell /bin/false harmony
mkdir -p /opt/harmony/bin /opt/harmony/uploads
mkdir -p /var/www/harmony

# Clonar el repositorio (como root)
git clone https://github.com/TU_ORG/harmony-v3.git /opt/harmony/repo

# Permisos
chown -R harmony:harmony /opt/harmony
chown -R www-data:www-data /var/www/harmony
ℹ
Si el repositorio es privado, agrega un Deploy Key de solo lectura en GitHub: ve a Settings → Deploy Keys → Add Deploy Key y pega la clave pública del servidor (cat ~/.ssh/id_rsa.pub o genera una nueva).
2
Crear el .env y la base de datos
# Crear .env (nunca se guarda en Git)
nano /opt/harmony/.env
# (completa igual que en el manual de Azure)
chmod 600 /opt/harmony/.env
chown harmony:harmony /opt/harmony/.env

# Crear base de datos PostgreSQL
sudo -u postgres psql -c "CREATE USER harmony WITH PASSWORD 'TuPasswordAqui';"
sudo -u postgres psql -c "CREATE DATABASE harmony_system OWNER harmony;"
sudo -u postgres psql -c "ALTER USER harmony CREATEDB;"

Paso 3 — Clave SSH para GitHub Actions

3
Generar un par de claves SSH dedicado para CI/CD

Ejecuta esto en el servidor de Azure (no en tu máquina local).

# Generar clave SSH sin passphrase para GitHub Actions
ssh-keygen -t ed25519 -C "github-actions-harmony" -f /root/.ssh/github_actions_deploy -N ""

# Agregar la clave pública al authorized_keys del servidor
cat /root/.ssh/github_actions_deploy.pub >> /root/.ssh/authorized_keys
chmod 600 /root/.ssh/authorized_keys

# Mostrar la clave PRIVADA (cópiala para el Paso 4)
cat /root/.ssh/github_actions_deploy
# -----BEGIN OPENSSH PRIVATE KEY-----
# b3BlbnNzaC1rZXktdjEAAAA...   ← copia TODO esto incluyendo las líneas "---"
# -----END OPENSSH PRIVATE KEY-----
✗
Nunca guardes la clave privada en Git. Solo va en los Secrets de GitHub, nunca en el repositorio.

Paso 4 — Configurar los Secrets en GitHub

4
Agregar los secrets en el repositorio de GitHub

Ve a tu repositorio → Settings → Secrets and variables → Actions → New repository secret.

Agrega estos 3 secrets:

Secret NameValor
SERVER_HOSTLa IP pública de tu VM de Azure (ej: 20.195.x.x)
SERVER_USERroot (o el usuario que tiene acceso SSH)
SSH_PRIVATE_KEYTodo el contenido del archivo /root/.ssh/github_actions_deploy
✓
Para pegar la clave privada: copia desde -----BEGIN OPENSSH PRIVATE KEY----- hasta -----END OPENSSH PRIVATE KEY-----, incluyendo esas líneas.

Paso 5 — El archivo del workflow

5
Crear el archivo de GitHub Actions en el repositorio

Crea este archivo en tu máquina local dentro del repositorio y haz commit:

.github/workflows/deploy.yml
name: Deploy Harmony v3

on:
  push:
    branches:
      - main

jobs:
  deploy:
    name: Deploy to Azure VM
    runs-on: ubuntu-latest
    timeout-minutes: 15

    steps:
      - name: Deploy via SSH
        uses: appleboy/ssh-action@v1.0.3
        with:
          host: ${{ secrets.SERVER_HOST }}
          username: ${{ secrets.SERVER_USER }}
          key: ${{ secrets.SSH_PRIVATE_KEY }}
          script: |
            bash /opt/harmony/deploy.sh 2>&1

Paso 6 — Script de despliegue en el servidor

6
Crear el script que ejecuta el pipeline

Crea este archivo en el servidor de Azure:

nano /opt/harmony/deploy.sh
/opt/harmony/deploy.sh
#!/bin/bash
set -e

export PATH=$PATH:/usr/local/go/bin
REPO_DIR="/opt/harmony/repo"
BIN_DIR="/opt/harmony/bin"
WEB_DIR="/var/www/harmony"

echo "=== [1/5] Actualizando código ==="
cd "$REPO_DIR"
git pull origin main

echo "=== [2/5] Compilando API Go ==="
cd "$REPO_DIR/api"
go mod download
go build -ldflags="-s -w" -o "$BIN_DIR/harmony-api-new" ./cmd/server/main.go
mv "$BIN_DIR/harmony-api-new" "$BIN_DIR/harmony-api"
chown harmony:harmony "$BIN_DIR/harmony-api"

echo "=== [3/5] Compilando Frontend React ==="
cd "$REPO_DIR/web"
npm ci --prefer-offline
npm run build

echo "=== [4/5] Actualizando archivos estáticos ==="
rm -rf "$WEB_DIR"/*
cp -r dist/* "$WEB_DIR/"
chown -R www-data:www-data "$WEB_DIR"

echo "=== [5/5] Reiniciando servicio ==="
systemctl restart harmony-api
systemctl status harmony-api --no-pager

echo "=== Deploy completado: $(date) ==="
chmod +x /opt/harmony/deploy.sh
✓
El script compila el nuevo binario como harmony-api-new primero, y solo lo reemplaza si compila sin errores. Esto evita que un error de compilación borre el binario funcional.

Configura Nginx exactamente como en el manual de Azure manual (misma configuración de proxy, WebSocket, SSL). Si ya lo tienes configurado, pasa al Paso 8.

# Crear config de Nginx (igual que en el manual de Azure)

Paso 9 — Verificar que el pipeline funciona

8
Hacer push y observar el pipeline

En tu máquina local, haz cualquier cambio, commit y push a main:

# En tu máquina local
git add .github/workflows/deploy.yml
git commit -m "ci: agregar GitHub Actions para deploy en Azure"
git push origin main

Ve a tu repositorio en GitHub → Actions. Deberías ver el workflow corriendo. Clic en él para ver los logs en tiempo real.

✓
El deploy tarda ~2-3 minutos (git pull + go build + npm build). Cuando el Action muestra ✓ en verde, el nuevo código ya está en producción.

Paso 10 — Rollback de emergencia

Si un deploy rompe la app, puedes revertir en segundos:

# Opción 1: Revertir en Git y hacer push (activa el pipeline con el commit anterior)
git revert HEAD
git push origin main

# Opción 2: Rollback manual directo en el servidor
ssh root@TU_IP_AZURE
cd /opt/harmony/repo
git log --oneline -5           # ver últimos commits
git checkout HASH_DEL_COMMIT_BUENO  # volver al commit anterior
bash /opt/harmony/deploy.sh    # recompilar y reiniciar
⚠
Después de un rollback manual con git checkout, recuerda que el servidor quedará en modo "detached HEAD". Para el siguiente deploy desde GitHub Actions, haz git checkout main && git pull primero.

Monitoreo post-deploy

# Estado del servicio
systemctl status harmony-api

# Logs en tiempo real
journalctl -u harmony-api -f

# Ver los últimos deploys (commits)
cd /opt/harmony/repo && git log --oneline -10
